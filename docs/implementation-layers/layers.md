# Implementation Layers

> Every concept in the AstraPlanner documentation classified into build phases.
> Rule: a concept moves to a later phase only when its prerequisite data, infrastructure, or validated assumptions do not yet exist.

---

## PHASE 1: MVP (Weeks 1-24)

Only what is needed for a single-site, working workforce planning tool that replaces spreadsheet-based shift planning.

## PHASE 2: Enhanced (Months 7-12)

Features that require accumulated data from Phase 1 operations, validated Phase 1 components, or infrastructure that depends on a working base product.

| Concept | Current Doc | What to Build | Prerequisite | Why This Phase (Not Earlier) | What Goes Wrong If Built Too Early | Dependency Chain |
|---------|------------|---------------|-------------|------------------------------|-----------------------------------|-----------------|
| **Absence prediction ML** | `learning-model.md`, `ai-evolution-roadmap.md` Phase 2 | Train statistical models on Phase 1 absence data; predict P(absent) per employee per day by day-of-week, season, team, recent history; surface as "Department B has 18% Monday absence probability" | 3+ months of absence/leave data logged in Phase 1; enough data to compute day-of-week baselines; AUC-ROC > 0.75 validation target | ML models require training data. At launch there is zero absence history in the system. Even the simplest day-of-week baseline needs 13 weeks of data. | Training on insufficient data produces models with poor accuracy that erode planner trust. A model that says "15% absence probability" when the actual rate is 5% causes overstaffing. Recovery from lost trust is harder than building trust from scratch. | Phase 1 data capture (3+ months) -> feature engineering pipeline -> model training -> validation -> deployment |
| **Recommendation engine (basic)** | `recommendation-engine.md`, `ai-evolution-roadmap.md` Phase 2 | Personalized staffing recommendations ranked by predicted relevance; cross-training suggestions based on skill concentration risk; weekly insight digest; recommendation fatigue management | User behavior data from Phase 1 (which suggestions accepted/rejected, override patterns); at least 5 active tenants with consistent usage data; accept/dismiss ratio baseline > 20% | Personalization requires observing how each planner uses the system. ## PHASE 3: Enterprise (Year 2+)

Features that require organizational maturity, accumulated learning data, regulatory clearance, or scale that justifies the complexity.

| Concept | Current Doc | What to Build | Prerequisite | Why This Phase (Not Earlier) | What Goes Wrong If Built Too Early | Dependency Chain |

## Decision Rules for Phase Assignment

1. **No data, no ML.** Any feature requiring training data belongs in Phase 2 at earliest (3+ months after launch).
2. **No users, no personalization.** Features that adapt to user behavior require Phase 1 interaction data.
3. **No validated solver, no advanced solver.** CP-SAT, GA, SA, and Monte Carlo all presuppose that the base HiGHS solver is working and trusted.
4. **No stable schema, no event sourcing.** Event sourcing amplifies the cost of schema changes. Wait until the schema is stable.
5. **No customer, no CBA.** Union/CBA rules are customer-specific. Building a generic engine without customers produces unusable code.
6. **No proven L1, no L2. No proven L2, no L3.** Automation levels cannot be skipped. Each level earns the trust needed for the next.
7. **Build one view, then decide on the second.** Three scheduling views built simultaneously means three half-finished views. Ship one, research the next.
8. **Infrastructure follows need, not vision.** Kafka, CRDTs, multi-region, Ruflo, and event sourcing solve problems that do not exist at launch.
